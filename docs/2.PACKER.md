# Baking AMIs

Ensure credentials in your AWS profile are refreshed, then bake an AMI with the [bake-ami.sh](../core-components/bake-ami.sh)
script:
```
$ ./core-components/bake-ami.sh
Usage:
  bake-ami.sh
    -a : [Required] account name
    -v : [Required] vpc name
    -t : [Required] ami type -> cassandra
    -i : base ami_id
```

* `-a` account name and `-v` vpc name are required; they must match directories under `configurations/`.
* `-t cassandra` produces an **Apache Cassandra 3.11.19** image.
* The base AMI can be provided in one of three ways:
  * With the `-i` option to pass a specific AMI ID.
  * As `PACKER_BASE_AMI_ID` in [variables.yaml](../configurations/default-account/variables.yaml).
  * If not specified, the script auto-discovers the latest **Amazon Linux 2023** AMI (filter: `al2023-ami-20*-kernel-*-x86_64`).
* **Important:** IMDSv2 is enforced (`HttpTokens=required`) on all instances. All metadata requests use session tokens — see the scripts for examples.
* **Important:** the `PACKER_SUBNET_ID` variable must be a **public subnet** with a route to an Internet Gateway. Otherwise, Packer will time out waiting for SSH.
* Java is installed via **Zulu** (`java_use_zulu: true`), which works on Amazon Linux 2023 (openjdk packages do not exist on AL2023).

## Networking during instance lifecycle

The baked AMI handles dual-ENI networking for Cassandra cluster communication:

**Amazon Linux 2023+ instances** (recommended):
- Secondary ENI is automatically brought up by `amazon-ec2-net-utils` with predictable `ensN` naming.
- Source-based routing is configured automatically — traffic from the secondary ENI egresses with its own IP (no asymmetric routing).
- The [enable_eth1.sh](../core-components/packer/cassandra/scripts/enable_eth1.sh) script uses IMDSv2 to discover the secondary interface and reports its IP.

**Amazon Linux 2 / RHEL 7 instances** (legacy support):
- Secondary ENI comes up as `eth1` using classic udev naming.
- Manual ifcfg-eth1 configuration is created and `ifup eth1` / `ifdown eth0` are called (assuming you're running over eth1 for inter-node traffic).
- The enable_eth1.sh script handles this path transparently if `/sbin/ifup` exists.

**IMDSv2 enforcement:**
All metadata queries (IMDS) require an X-aws-ec2-metadata-token session token header. This is enforced at launch time via `HttpTokens=required`. See [gen_server_keystores.sh](../core-components/packer/cassandra/scripts/gen_server_keystores.sh) and [fetch-credentials.sh](../core-components/packer/fetch-credentials.sh) for examples.

## SSL/TLS certificate generation

When `AUTO_START_CASSANDRA=1` (Terraform templates set this), the `gen_server_keystores.sh` script is called during bootstrap with:
```
gen_server_keystores.sh <tfstate_bucket> <artifact_path> <account_name> <vpc_name> <cluster_name> <region> <private_ip>
```

The script expects:
- Secrets (keystore/truststore passwords) in AWS Parameter Store at `/cassandra/<account>/<vpc>/<cluster>/secrets/*`.
- Optional pre-baked CA cert/key in S3 at `s3://<bucket>/<account>/<vpc>/<cluster>/files/certs/` and `files/keystores/`.
- **IMDSv2 token** for all metadata requests (enforced).

If you have pre-existing certificates, upload them to S3 before deploying (see [1.INITIAL_SETUP.md § 8. Setup - SSL cert](1.INITIAL_SETUP.md#8-setup---ssl-cert)).

## Known issues

> `Error launching source instance: PendingVerification: Your request for accessing resources in this region is being validated...`

A brand new AWS account will sometimes return this error. Just wait it out and try again later.

> 404 error while running bake-ami.sh

The script uses `curl -s -4 ifconfig.co` to get your external IP address (for use in a security group's inbound rules). Sometimes ifconfig doesn't respond. Just run the command again.

> `Another app is currently holding the yum lock; waiting for it to exit...`

This happens (often) at the start of a Packer build. Just wait it out, the yum lock should be cleared after a few seconds.
