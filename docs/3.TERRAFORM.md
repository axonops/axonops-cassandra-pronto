# Deploying

Ensure credentials in your default profile are refreshed and ready to go, then deploy AWS resources with the [terraform.sh](../core-components/terraform/terraform.sh) script in the `core-components/terraform` directory.

The script creates a temporary build directory with all required variables, initializes Terraform, and runs the requested action. Use the `-t` option to specify `plan` first (to [review what will be deployed](https://www.terraform.io/docs/commands/plan.html)), then `apply`.

```
$ ./core-components/terraform/terraform.sh
Usage:
  terraform.sh
    -a : account name
    -v : vpc name
    -c : cluster name
    -l : layer -> account-resources, vpc-resources, cluster-resources
    -t : action -> plan, apply, show
    -d : credentials file
    -r : [Optional] name of assumed_role (cross-account only; omit for same-account deploys)
    -i : account id
    -b : tfstate bucket name
    -o : ami owner acct id
    -k : keep build dir after terraform runs
    [-x <extra args>]
```

Deploy in this sequence:

1. **Account layer** (`-l account-resources`): deploys IAM roles and cross-account permissions.
2. **VPC layer** (`-l vpc-resources`): depending on `TERRAFORM_MANAGED_VPC` in [variables.yaml](../configurations/default-account/variables.yaml), either creates a new VPC or reads an existing one. Also deploys shared VPC resources (bastion instance, NAT gateway for private subnet access).
3. **Cluster layer** (`-l cluster-resources`): deploys the Cassandra cluster resources (EC2 instances, auto-scaling groups, security groups).

**Key changes from previous versions:**
- The `-r` (assumed_role) flag is now **optional** — omit it entirely for same-account deploys; include it only for cross-account STS AssumeRole scenarios.
- Provider versions are pinned: AWS ~> 6.53, cloudinit ~> 2.4, Terraform >= 1.5 and < 2.0.
- Terraform now uses `aws_launch_template` (replacing deprecated `aws_launch_configuration`) and `templatefile()` function with the cloudinit provider (replacing `template_file` and `template_cloudinit_config` data sources).
- Data source `aws_subnets` replaces deprecated `aws_subnet_ids`.

## VPC Networking

When `TERRAFORM_MANAGED_VPC=false` (reusing an existing VPC), a NAT Gateway is automatically provisioned to provide the private "Data" subnets with outbound AWS API access. This is necessary because instances in these subnets have no public IP and would otherwise have no route to AWS services (SSM Parameter Store, EC2, S3, STS, etc.).

The NAT Gateway is scoped to avoid modifying any shared main route table; a dedicated route table is created and associated only with the data subnets.

## Security Groups

The `sg_ops_nodes_to_cas` output from the vpc-shared module is currently stubbed to an empty string. This allows future "ops node" access security groups to be attached to Cassandra instances without blocking cluster provisioning. If you need to grant ops nodes access to Cassandra ports, see [PENDING_DECISIONS.md](../docs/PENDING_DECISIONS.md).

## Known Issues

> `Error loading state: InternalError: We encountered an internal error. Please try again.`

Sometimes when deploying for the first time, this error appears between Terraform commands. Retry the command.

> `aws_ssm_parameter.parameter: value of 'count' cannot be computed`

Terraform has an issue calculating the size of lists passed to [dynamic resources](../core-components/terraform/modules/parameter-store/parameters.tf) if the list contains any elements that are interpolated while the plan is being assembled. The workaround is to specify a static list size.

### SSH

After deploying the `account` and `cluster` layers (producing bastion & cassandra nodes) you'll be able to SSH.  Your Cassandra
nodes should end up with a second EIP attached, after the bootstrap process has finished:

![IP](./images/cassandra_ips.png)

If not, check the "bootstrap debugging" section [here](MORE_DETAILS.md).

The second EIP is the address you'll want to SSH to.  First, create ssh_config with the [build-ssh-config.sh](../core-components/scripts/ssh/build-ssh-config.sh)
script:
```
$ ./core-components/scripts/ssh/build-ssh-config.sh
Usage:
  build-ssh-config.sh
    -a : [Required] account name
    -v : [Required] vpc name
    -c : [Required] cluster name
```
Then SSH using the generated config:
```
$ ssh -F ssh_config ${IP_ADDRESS}
```
