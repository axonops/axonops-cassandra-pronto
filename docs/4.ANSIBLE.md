# Runtime Operations

## Prerequisites

Some of the prep for Ansible requires awscli, Python3, and supporting libraries. If running on macOS, [see Python 3 installation instructions](https://docs.python-guide.org/starting/install3/osx/):

```bash
brew install awscli python3
python3 -m pip install --user boto3 tornado pyyaml six
```

If a previously installed "six" library has root permissions and causes an error:

```bash
sudo chown -R ${YOUR_USERNAME} $(python -m site --user-site)
python3 -m pip install --user --ignore-installed six
```

## Waiting for Cloud-Init

After deploying your cluster, **wait 3–5 minutes** before running remote Ansible operations. The AMI runs cloud-init scripts that attach the secondary elastic network interface (ENI) and configure it as the `active_interface`. Ansible cannot reach nodes until cloud-init has completed.

## Running Operations Playbooks

Runtime operations for Cassandra tend to be cluster- and application-specific. A few example playbooks are provided and can be executed via the [operations.sh](../core-components/operations.sh) script:

```bash
$ ./core-components/operations.sh
Usage:
  operations.sh
    -a : [Required] account name
    -v : [Required] vpc name
    -c : [Required] cluster name
    -o : [Required] operation
    -h : host ip (default: all)
```

The script performs these steps:

1. **Discovers nodes**: Finds the bastion load balancer DNS name and active IP addresses of your cluster, then builds an `ssh_config` file for Ansible (and manual SSH).
   - The SSH config generator is available separately at [build-ssh-config.sh](../core-components/scripts/ssh/build-ssh-config.sh).
   - To SSH manually: `ssh -F ./ssh_config <ip_address>` (use the active secondary ENI after Terraform completes).

2. **Builds inventory**: Creates a host list for Ansible to consume.

3. **Runs the playbook**: Executes the operation specified via `-o`.
   - Available operations: `init`, `start`, `stop`, `restart`, `mount`, `unmount`, `restack`.
   - Playbooks are located in [core-components/ansible/playbooks](../core-components/ansible/playbooks).

## Cassandra AMI Baking

The Cassandra AMI is built with Packer using the [bake-cassandra.yml](../core-components/packer/cassandra/ansible/bake-cassandra.yml) playbook, which installs Java, Apache Cassandra, and the AxonOps agent package via the `axonops.axonops` Ansible collection.

**Key variables:**
- `cassandra_version`: Set to `{{ cassandra_full_version }}` — the specific Apache Cassandra version (e.g., 3.11.x or 4.x).
- `java_use_zulu: true`: Selects Amazon Corretto Zulu for Java (required for compatibility with AL2023 base image).
- `cassandra_start_on_boot: false`: Cassandra does not start automatically; runtime config (datacenter, rack, seed list, listen address) is applied at instance boot by the bootstrap script.

Per-node runtime configuration is applied at instance launch time via [cassandra-init.tpl](../core-components/terraform/modules/cassandra/files/cassandra-init.tpl) cloud-init template and `/opt/cassandra/scripts/bootstrap.sh`.

## Verifying Cluster Health

After the cluster has stabilized (3–5 minutes post-deployment), SSH to a Cassandra node and run:

```bash
nodetool status
```

Expected output (3-node cluster):

```
Datacenter: us-west-2a
======================
Status=Up/Down
|/ State=Normal/Leaving/Joining/Moving
--  Address     Load       Tokens       Owns    Host ID                               Rack
UN  10.0.1.100  404.28 KiB  256          33.3   95ba1239-94cc-4809-9535-f6f620f54122  us-west-2a
UN  10.0.2.101  403.35 KiB  256          33.3   54f13ae9-bf55-485a-9dfb-663daa7b0a79  us-west-2b
UN  10.0.3.102  371.37 KiB  256          33.3   1710316c-33f0-4a81-befa-7504a6e41592  us-west-2c

Note: Non-system keyspaces don't have the same replication settings, effective ownership information is meaningless
```

All nodes should show `UN` (Up/Normal). If any node shows `DN` (Down), wait another minute for that node to complete bootstrap.
